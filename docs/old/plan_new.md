# 架构重构新计划（留档版）

> 适用范围：`GameEngine`、`DMAgent`、`StateEvolution`、`NPC` 决策链路、提示词与上下文管理  
> 目标：修正上一版 proposal / execution plan 的颗粒度问题，锁定可执行边界，避免继续出现实现偏航

## 1. 背景与问题定义

上一版计划的问题不在于“方向完全错误”，而在于**关键决策没有被写死**，导致实现者可以对同一目标做出不同理解，进而出现以下偏差：

1. NPC 相关逻辑已经部分落地，但实现与文档不一致，尤其是 `NPCDirector` 当前仍是规则兜底而不是 LLM 主导。
2. 行动优先级规则没有被明确写成不可变规范，导致出现使用归一化比值计算、而非原始属性值排序的问题。
3. `queue` / `reactive` 的关系没有被明确为“兼容标签”还是“并行架构”，从而让实现继续朝双轨分叉。
4. NPC 模块归属、prompt 归属、fallback 规则都没有在计划中锁定，后续代码很容易继续硬编码。

这份新计划的目标不是再扩大抽象层，而是：

- 明确哪些行为必须改
- 明确哪些行为只允许保留兼容
- 明确哪些行为是迁移期临时措施
- 明确哪些行为已经禁止继续分叉

## 2. 现状严重程度判断

### 2.1 P0：会直接导致架构偏航的点

- `NPCDirector` 目前仍然是规则兜底逻辑，如果继续沿用，会把“LLM 导演”这个核心承诺落空。
- 行动优先级如果继续使用归一化比例值，会直接影响回合顺序，属于静默错误。
- `queue` / `reactive` 若继续作为两套独立流程推进，会让新旧逻辑长期共存，最终很难收敛。

### 2.2 P1：会持续放大返工的点

- NPC 模块如果不迁移到 `src/agent/` 体系下，后续 prompt、LLM、状态推演的边界会继续混在一起。
- prompt 如果继续写死在 Python 中，任何改动都要改代码，且难以审阅与留档。
- 兼容字段如果没有明确“兜底优先级”，就会出现到底以结构化字段为准还是以旧字段为准的争议。

### 2.3 P2：表达不清但尚可补救的点

- NarrativeContext 已经出现基础实现，但需要继续收敛窗口、摘要、关键事实的职责边界。
- 旧文档里关于“方案一 / 方案二”的描述不够稳定，已经需要在新计划中直接定稿。

## 3. 总体决策

### 3.1 统一行动流程采用“方案二”

本次计划明确采用**方案二**作为当前实现路线：

- 先执行玩家主流程
- 再在同一回合上下文内统一处理 NPC 响应
- `queue` 和 `reactive` 只保留为兼容标签和触发来源，不再作为两套并行架构继续开发

选择方案二的原因：

- 实现更简单
- 时序问题更少
- 更贴近当前代码实际形态
- 更容易分阶段验收

### 3.2 方案一的处理方式

方案一保留为**概念层接口**，不作为本次迭代的落地目标：

- 可以在文档和模型命名中保留“统一队列”这一抽象概念
- 但不要求先构建完整的前置 NPC 行动队列再执行玩家回合
- 后续若要升级为方案一，必须另起明确重构，不允许在当前版本里混搭实现

### 3.3 NPC 导演必须走 LLM

NPCDirector 的正式目标是：

- 输入：`GameState`、玩家意图、最近事件、上下文
- 输出：结构化的 `NPCActionDecision` / `NPCActionForm`
- 决策主体：LLM
- 规则逻辑：只允许保留为安全护栏与兜底，不允许充当主决策树

## 4. 关键设计约束

### 4.1 优先级规则

行动队列优先级必须显式写成以下规则：

1. 先过滤不可行动角色
2. `hp <= 0` 的角色不入队
3. `san <= 0` 的角色不入队
4. 对可行动角色，按原始 `DEX` 值降序排序
5. 再按 `hp_ratio` 降序排序
6. 再按 `san_ratio` 降序排序
7. 最后用稳定 tiebreaker 保证顺序确定性

禁止事项：

- 禁止把 `DEX` 先除以 100 再参与排序
- 禁止把 hp / san 作为可行动角色的“加权加成”而不是过滤条件
- 禁止在不同分支里使用不同的优先级公式

### 4.2 模块归属

NPC 相关代码必须逐步迁移到 agent 体系内：

- 新路径建议：`src/agent/npc/`
- 旧路径 `src/npc/` 仅允许作为迁移期兼容层
- 最终目标是让 NPC 决策、DM、StateEvolution 都位于同一套 agent 组织方式下

### 4.3 Prompt 归属

所有用于 NPC 决策的提示词必须外置：

- 不允许写死在 Python 字符串里
- 不允许把导演提示、NPC 行为格式、fallback 提示混为一段硬编码
- 建议沿用现有 `src/agent/prompt/` 的 markdown 组织方式

### 4.4 兼容策略

迁移期保留以下兼容：

- `DMAgentOutput` 中的旧 NPC 字段
- `npc_intent` 兜底
- `queue` / `reactive` 世界配置
- 旧存档的加载与恢复

但兼容策略必须明确：

- 新结构化数据优先
- 旧字段只做兜底
- 兼容层不能反向污染新主流程

## 5. 推荐实施路径

### Phase 0：收束规格

目的：先把“怎么做”写死，再继续改代码。

- 固化方案二作为当前唯一落地路线
- 固化优先级公式
- 固化 NPCDirector 的职责边界
- 固化 prompt 外置规则
- 固化兼容优先级

交付物：

- 更新后的计划文档
- 一份明确的不变量清单

验收标准：

- 后续实现者不会再对“方案一还是方案二”“是否必须 LLM”“优先级怎么算”产生歧义

### Phase 1：修正队列与行动可行性

目的：先修正会直接影响回合排序和行动资格的逻辑。

- 把优先级计算改成显式规则
- 保持 `hp <= 0` / `san <= 0` 的硬过滤
- 为队列排序增加确定性测试
- 为不可行动过滤增加回归测试

验收标准：

- 排序结果可解释、可预测、可测试
- 不会再出现把低属性角色错误排到前面的情况

### Phase 2：抽出 NPCDirector 主决策层

目的：让 NPC 决策从引擎内部规则树中脱离出来。

- 建立 LLM 驱动的 NPCDirector
- 输出统一的结构化计划
- 保留最小化安全兜底逻辑
- 引擎只负责调用与承接，不负责主决策
- 仿照目前实现的state_evolution.md文件中关于npc的提示词撰写提示词
- 仿照目前实现的state_evolution.py文件中关于npc的输入上下文确定给导演agent的上下文信息
- 上下文信息必须包含必要的系统整体信息,让其知道自己在做什么

验收标准：

- NPC 行为通过结构化结果进入后续链路
- 不再依赖 if/else 决定 NPC 主意图


### Phase 3：迁移模块与提示词

目的：让代码边界和提示词边界对齐。

- 把 NPC 代码迁移到 `src/agent/npc/`
- 把 NPC prompt 抽到 markdown 文件
- 统一加载方式
- 保留旧路径转发，避免立刻破坏已有调用

验收标准：

- prompt 可单独审阅
- NPC 逻辑不再散落在多个目录

### Phase 4：统一 runtime 流程

目的：把方案二落成稳定可运行的主链路。

- 玩家输入先走 DM / rule / state evolution
- 若需要 NPC 响应，再进入统一的 NPC 响应步骤
- 让 `queue` / `reactive` 成为语义标签，而不是双套实现

验收标准：

- 主流程可稳定跑通
- NPC 响应时序清晰
- 不再出现两套行为逻辑互相抢控制权

### Phase 5：回归与收口

目的：确认迁移没有破坏现有世界与存档。

- 补齐回归测试
- 验证旧世界配置仍能加载
- 验证旧存档仍能恢复
- 验证新结构化路径覆盖核心场景

验收标准：

- 旧配置可运行
- 新逻辑可验证
- 迁移期兼容层可以继续存在，但不影响主流程

## 6. 测试清单

### 6.1 优先级测试

- 验证原始 `DEX` 排序优先于比例值
- 验证 `hp <= 0` 和 `san <= 0` 直接跳过
- 验证排序在同分条件下稳定

### 6.2 NPCDirector 测试

- 验证输出是结构化计划而不是纯文本
- 验证 LLM 失败时有安全兜底
- 验证不会继续走规则树主决策

### 6.3 流程测试

- 验证玩家主流程先执行
- 验证 NPC 响应在同一回合上下文内发生
- 验证变化写回状态后不会破坏后续流程

### 6.4 兼容测试

- 验证旧世界配置可读
- 验证旧存档可恢复
- 验证旧字段仍能兜底

### 6.5 Prompt 测试

- 验证 prompt 从 markdown 文件加载
- 验证 prompt 缺失时能明确失败
- 验证 prompt 内容不再硬编码在 Python 中

## 7. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| LLM 导演输出不稳定 | 中 | 保留结构化 schema 校验与兜底动作 |
| 兼容层过长 | 中 | 只保留迁移期必要转发，不扩展新功能 |
| 队列语义再度分叉 | 高 | 将方案二写入正式规范，禁止继续开新分支 |
| 提示词再次硬编码 | 高 | prompt 文件化，代码只负责加载 |
| 旧存档加载失败 | 高 | 新旧字段并存一段时间，先保证读取兼容 |

## 8. 明确不做什么

这份计划明确不做以下事情：

- 不做一轮内完整前置统一队列的方案一落地
- 不做 `GameEngine` 的一口气重写
- 不做 NPC、Narrative、StateEvolution 的大一统式重构
- 不立即删除所有旧字段和旧路径
- 不把兼容期和最终态混为一谈

## 9. 最终结论

这次重写的核心不是“再画一个更大的蓝图”，而是把已经混乱的决策颗粒度重新收拢。

当前建议是：

- 以**方案二**作为落地路线
- 以**LLM 导演 + 结构化输出**作为 NPC 主决策方式
- 以**原始 DEX + hp/san 硬过滤**作为队列规则
- 以**模块归属清晰、prompt 外置、兼容保留**作为迁移原则

这样做的好处是：

- 风险更低
- 时序更清晰
- 开发阻力更小
- 更适合作为当前版本的可执行计划
