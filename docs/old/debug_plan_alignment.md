## 3. 调试颗粒度定义（统一标准）

每个问题按 4 层逐层定位，避免一次性大改：
1. 入口层：输入如何进入流程（InputSystem / process_input）。
2. 决策层：DM/StateEvolution 的输出是否符合约束。
3. 应用层：StateChange 在 IOSystem 是否被正确、原子地应用。
4. 展示层：CLI/状态文本是否与真实状态一致。

每个问题都要求同时给出：
- 最小复现输入（1-3 条玩家输入）。
- 断言点（状态字段 + 用户可见输出）。
- 回归测试落点（tests 下的目标文件）。

## 4. 分问题 Debug 计划

## 4.1 架构建议整合（来自用户建议）

### 建议内容（已采纳到计划）
- 中期方向：弱化原有 NPC 自主行动队列，改为“响应式 NPC”。
- 新增 DM-NPC 调用通路：由 DM 视角扮演 NPC 进行回应，避免信息可见性硬编码。
- NPC 激活由 DM Agent 决策，而不是固定时序触发。

### 分阶段落地（避免一次性重构）
1. 第一阶段（本轮修复）：保留当前流程，先修复位置、移动、非法 ID、温度统一。
2. 第二阶段（架构升级）：引入 DM-NPC 通路并逐步替换队列机制。

### 目标流程（第二阶段）
玩家输入 -> DM Agent -> (需要鉴定)规则系统 -> 状态推演 -> (需要NPC回应)DM-NPC -> (需要鉴定)规则系统 -> 状态推演 -> 返回叙事

### 验收要点
- NPC 只在“需要回应”时触发，避免每回合额外调用。
- 叙事时序与跑团体验一致，且可通过测试稳定复现。


## 4.2 问题B：行动时序不一致（问题4）

### 假设
当前“玩家前置 NPC 行动”策略与叙事预期冲突，导致体感错位。

### 检查点
- `process_input` 中 NPC 前置调用时机。
- 对话输入与自然语言行动输入是否同一时序策略。

### 最小复现
1. 设置队列为 NPC 在前、玩家在后。
2. 输入“我认输”或“我先不动”。
3. 观察输出叙事顺序与状态变化顺序是否一致。

### 修复验收
- 明确单一时序策略并落地（文档化）：
  - 要么始终“玩家输入前最多一个 NPC 前置行动”；
  - 要么调整为“玩家后置触发 NPC”。
- 策略与测试保持一致，不出现命令输入误触发 NPC。

### 结论
- 支持该方向，建议作为第二阶段架构改造执行。
- 本轮先处理确定性缺陷，避免与重构耦合导致回归范围过大。

## 4.3 问题C：非法物品 ID 注入背包（问题7）
### 用户建议（已纳入）
- 增加 move 指令，优先支持 `to=<map_id>` 的直接目标写法，并兼容原有输入。
- 简化 Agent 输出表单：尽量只输出 `id + operation + value` 等必要字段。
- 复杂一致性与约束尽可能下沉到 IOSystem，提升容错并让 erro 反馈更稳定。
### 假设
StateEvolution 可能输出不存在的 item_id，当前仅部分校验。

### 检查点
- `src/agent/state_evolution.py`：`validate_changes` 是否覆盖 inventory ADD/UPDATE 合法性。
- `src/data/io_system.py`：`apply_state_change` 是否对 inventory 写入做实体存在性拦截。

### 最小复现
1. 构造 StateChange：向玩家背包 ADD `item-not-exist-01`。
2. 执行应用后查看玩家背包与返回码。

### 修复验收
- 决策层拦截：非法物品变更在进入应用层前被拒。
- 应用层兜底：即使漏过决策层，IO 仍拒绝非法 ID。
- 背包展示不再出现“未知物品”由系统自身错误导致的脏数据。
- 补测试：新增 inventory 非法 ID 拒绝用例。
- Agent 输出协议在不破坏兼容性的前提下收敛为简表单。

## 4.4 问题D：温度配置不统一（问题8）

### 假设
DM/StateEvolution 里存在硬编码温度，覆盖配置文件。

### 检查点
- `src/agent/dm_agent.py` 与 `src/agent/state_evolution.py` 的 `call_llm_json(... temperature=...)`。
- `config/llm.json` 与环境变量优先级是否统一。

### 最小复现
1. 将 `config/llm.json` 设为固定值（如 0.0）。
2. 跑一次 DM 与 StateEvolution 调用。
3. 断言实际调用参数与配置一致。

### 修复验收
- 无硬编码温度覆盖（除非明确允许并集中管理）。
- 温度来源可追踪：环境变量 > 配置文件 > 默认值。
- 补测试：mock LLMService，断言传参一致。

## 4.5 问题E：移动功能缺失（问题9）

### 假设
缺少显式 move 指令，纯依赖自然语言推演导致位置变更不稳定。

### 检查点
- `src/agent/input_system.py` 是否支持 `\\move`/`\\go`（或中文别名）。
- 场景邻接校验是否存在（目标是否为 current_map.neighbors）。

### 最小复现
1. 在图书馆输入移动到走廊。
2. 立即执行 `\\where` 与 `\\look`。
3. 断言位置与场景一致。

### 修复验收
- 提供可预测的基础移动指令（推荐）。
- 非法目标（非邻接）返回明确提示，不改状态。
- 补测试：合法移动 + 非法移动两类。

## 4.6 问题F：位置系统不一致（问题10）

### 假设
`get_current_map` 只依赖 `current_scene_id`，与 `player.location` 可分叉。

### 检查点
- `src/data/models.py` 中 `get_current_map` 的取值策略。
- 所有改变玩家位置的路径是否同步更新 `current_scene_id`。

### 最小复现
1. 仅更新 `player.location` 为走廊，不更新 `current_scene_id`。
2. 执行 `\\status` / `\\where`。
3. 观察显示位置是否仍停留旧场景。

### 修复验收
- 单一真相源策略二选一并固定：
  - 方案1：以 `player.location` 为真相源，`current_scene_id` 为缓存并强制同步；
  - 用户建议：采用方案1，并逐步弱化 `current_scene_id` 的业务含义。
  - 方案2：以 `current_scene_id` 为真相源，任何移动必须改它。
- 全链路一致：显示、DM 上下文、规则目标都使用同一位置语义。
- 补测试：位置字段一致性断言。
