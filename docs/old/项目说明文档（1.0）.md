1. 项目概述

1.1 项目背景与动机

在当前的语文教育与经典阅读中，中学生和大学生普遍面临阅读兴趣不足、理解深度不够的问题。传统电子书阅读器仅提供文本呈现，缺乏交互性与沉浸感，难以激发学生对名著的深入探索。同时，随着大语言模型（LLM）技术的发展，AI能够理解文本并生成结构化的内容，为构建“可进入的书中世界”提供了可能。

1.2 项目目标

本项目旨在开发一款名为 “书境Lab” 的Web应用，通过AI技术自动从名著文本中抽取关键信息，构建可交互的“书境场景”，提升用户的阅读兴趣与理解深度。项目将参加中国大学生计算机设计大赛，突出“人工智能应用”与“教育软件”的结合。

1.3 核心概念

书境Lab：AI驱动的名著沉浸式阅读实验室
基于大语言模型和Web交互技术，从名著文本中自动生成可探索的“书中世界”，通过互动化的阅读体验，让用户“走进”文学经典。

1.4 教育价值与大赛策略

· 教育价值：帮助学生在沉浸式体验中理解人物、情节与文化内涵，培养深度阅读习惯。
· 大赛策略：
· 紧扣“传统文化+沉浸场景+教育价值”的高频获奖组合；
· 强调AI应用的实际落地，而非单纯炫技；
· 突出项目的教育闭环：阅读前导入、阅读中支撑、阅读后反思。

1.5 技术实现概述

本项目基于 Python 技术栈构建沉浸式名著阅读平台。后端采用 FastAPI 作为 Web 框架，结合 Pydantic 定义“场景—人物—物品—知识点”等核心数据模型，通过官方 Python SDK 调用大语言模型，实现从书本文本到结构化场景数据与教学任务的自动抽取与生成。数据持久化使用 SQLite + SQLAlchemy 统一管理用户学习进度和预设书籍资源。
前端在初期采用 Streamlit 框架快速搭建交互界面，实现文本输入、场景卡片展示与互动问答；后期在此基础上嵌入 Three.js 页面，实现 Web 端的 3D/2.5D 场景展示。整体技术路线以 Python 为主，层次清晰、上手门槛低，既适合大一学生快速开发迭代，又能在中国大学生计算机设计大赛中清晰呈现“AI 场景引擎 + 教育交互”的技术与教学创新点。

2.总体设计理念

2.1 从COC文字冒险框架到“书境Lab”的迁移

本项目设计灵感来源于团队前期研究的 COC文字冒险游戏框架（见文档《COC文字冒险游戏框架Spec V2简化版》）。该框架的核心思想是：将权力交给AI，减少硬编码逻辑，通过“文字数据 ↔ 结构化数据”的双轨制，实现高自由度的叙事生成。我们将这一思想迁移至教育阅读场景，改造为面向名著的“书境体验馆”。

2.2 核心理念：AI驱动 + 结构化数据双轨制

· 文字数据：名著原文、AI生成的解释与提示文案。
· 结构化数据：场景元信息（时间、地点、氛围）、场景内元素（人物、物品、地点节点）。
· AI职责：负责文字到结构的解析（如从文本抽取场景信息），以及结构到文字的生成（如根据用户交互生成反馈）。

2.3 设计原则

· 模块化：各功能模块（LLM调用、数据持久化、交互界面）解耦，便于团队分工与后期扩展。
· 渐进式实现：初期用Python快速构建可演示的MVP，后期逐步增强3D交互。
· 类型安全：使用Pydantic严格定义数据结构，确保AI输出可靠性。

3.3 数据流说明

1. 用户选择或粘贴文本 → 前端POST /api/generateScene。
2. 后端调用LLM服务解析文本，生成场景结构化数据，存入数据库。
3. 返回场景数据给前端，前端渲染卡片。
4. 后端调用教学任务生成模块，结合用户交互生成反馈与下一步引导。
5. 状态推演模块更新用户进度，返回反馈内容。
6. 前端展示反馈，并更新界面。

4.数据模型

4.1 实体类型与结构

借鉴COC框架的实体设计，我们定义以下核心实体：

场景（Scene）

```json
{
  "id": "scene-001",
  "title": "林黛玉进贾府",
  "scene_type": "庭院",
  "time": "傍晚",
  "place": "贾府内院",
  "atmosphere": "庄重中带着几分压抑",
  "characters": ["char-001", "char-002"],
  "items": ["item-001", "item-002"],
  "learning_goals": ["理解林黛玉初入贾府的心理", "识别贾府的人物关系"],
  "knowledge_points": ["《红楼梦》背景", "封建社会礼仪"]
}
```

人物（Character）

```json
{
  "id": "char-001",
  "name": "林黛玉",
  "role": "主角",
  "description": {
    "public": [{"description": "一位体弱多病、眉目如画的少女"}],
    "hint": "她此时内心敏感，对贾府的奢华既新奇又不安"
  },
  "location": "scene-001"
}
```

物品（Item）

```json
{
  "id": "item-001",
  "name": "玉佩",
  "description": {
    "public": [{"description": "一块雕刻精美的玉佩"}],
    "hint": "这是贾母送给黛玉的见面礼，象征着家族接纳"
  },
  "location": "scene-001"
}
```

用户进度（UserProgress）

```json
{
  "user_id": "user-123",
  "scene_id": "scene-001",
  "status": "completed",  // "locked", "unlocked", "completed"
  "answers": [
    {
      "question_id": "q-001",
      "user_answer": "A",
      "is_correct": true,
      "feedback": "很好，你理解了..."
    }
  ],
  "score": 85
}
```

4.2 二级描述系统（public + hint）

· public：向用户展示的公开描述，可由AI在后续交互中追加。
· hint：仅供AI使用的提示信息（如人物心理、隐藏线索），用于生成更有深度的反馈，绝不直接展示给用户。

4.3 场景结构化表示

场景数据结构采用Pydantic模型定义，确保类型安全：

```python
from pydantic import BaseModel
from typing import List, Optional

class Character(BaseModel):
    id: str
    name: str
    role: str
    desc: str

class Item(BaseModel):
    id: str
    name: str
    meaning: str

class Scene(BaseModel):
    id: str
    title: str
    scene_type: str
    time: str
    place: str
    atmosphere: str
    characters: List[Character]
    items: List[Item]
    learning_goals: List[str]
    knowledge_points: Optional[List[str]] = []
```

4.4 数据存储方案

· 开发阶段：使用SQLite数据库，通过SQLAlchemy ORM操作。
· 表设计：
· scenes：预置场景基本信息
· characters、items：实体表，外键关联场景
· users：用户信息（可选）
· user\_progress：用户学习进度
· user\_answers：答题记录

5.核心模块详细设计

5.1 LLM服务模块

封装对大语言模型（如GPT、文心一言）的调用，提供统一接口。

主要函数：

· call\_llm(prompt: str, response\_format: dict) -> dict：发送prompt，要求模型以指定JSON格式返回。
· 包含重试机制（最多3次），若解析失败则降级返回默认内容。

Prompt设计示例（场景解析）：

```
你是一个文学场景解析助手。请根据以下文本片段，提取场景信息并以JSON格式返回。
文本：{text}
要求JSON包含：标题、时间、地点、氛围、人物列表（每个人物有姓名、角色、简短描述）、物品列表（每个物品有名称、象征意义）、学习目标（列表）。
确保输出合法的JSON对象。
```

5.2 场景解析引擎

输入：文本片段
输出：Scene对象（经Pydantic校验）

处理流程：

1. 构造Prompt调用LLM服务。
2. 解析返回的JSON，若字段缺失或格式错误，尝试修正或重新调用。
3. 返回校验通过的Scene对象，并存入数据库（若为新场景）。


5.3 状态推演与进度管理

· 维护每个用户对每个场景的访问状态（未解锁、进行中、已完成）。
· 记录答题情况，计算得分，生成个性化反馈。
· 当用户完成场景任务后，更新进度，推荐关联片段。

反馈生成：根据用户答题正确与否，结合场景hint信息，调用LLM生成鼓励性或纠正性的评语。

5.5 IO系统与数据持久化

提供对数据库的统一访问接口，隔离底层实现。主要方法：

· get\_scene(scene\_id)
· save\_user\_answer(user\_id, scene\_id, question\_id, answer)
· update\_user\_progress(user\_id, scene\_id, status)
· 等

5.6 用户交互接口（Input系统 + 前端）

前端通过REST API与后端通信。主要API端点：

· POST /api/generateScene：接收文本，返回场景数据。
· POST /api/submitAnswer：接收答题数据，返回评语及下一步提示。
· GET /api/getSceneList：获取可用的场景列表。
· GET /api/getUserProgress：获取当前用户进度。

6.技术实现方案

6.1 技术栈总览

层级 技术 说明
后端框架 FastAPI 高性能异步框架，自动生成OpenAPI文档
数据校验 Pydantic 定义数据结构，保证类型安全
AI调用 OpenAI SDK  调用云端大模型
数据库 SQLite + SQLAlchemy 轻量级，便于开发与部署
前端（初期） Streamlit 纯Python快速搭建Web界面

6.2 后端：FastAPI + Pydantic + SQLite

· 使用FastAPI构建RESTful API。
· 用Pydantic定义请求/响应模型。
· SQLAlchemy作为ORM，定义模型类。

6.3 AI调用：云端大模型API

· 选择国内可用的模型,以保证稳定性。
· 封装统一的LLM客户端，支持切换模型。
· Prompt设计需明确要求输出JSON，并给出示例。

6.4 前端分阶段实现

Streamlit（2-3周）

· 搭建基础界面：左侧文本输入/选择，中间场景卡片，右侧人物/物品卡片，下方题目区。
· 使用st.session\_state管理状态，通过requests调用后端API。



6.5 部署方案

· 后端：使用uvicorn运行FastAPI应用，Nginx反向代理。
· 前端：Streamlit应用可直接部署（需开放端口
· 数据库：SQLite文件随应用部署，定期备份。

7.教育功能设计

7.1 阅读前：兴趣唤起

· 通过生活化问题引入，例如：“如果你穿越到《红楼梦》贾府，你最想见谁？”
· 展示场景概览（时间、地点、人物），激发好奇。


7.2 阅读后：反思与迁移

· 系统生成1-2个思考题（如“如果你是林黛玉，你会如何应对？”）。
· 推荐关联片段（如“接下来可以阅读‘宝玉摔玉’场景”）。


8.用户交互流程与界面设计

8.1 用户体验流程

1. 选择文本：从内置书单选择或粘贴自定义文本。
2. 生成场景：点击“进入书境”，系统解析并展示场景。
3. 探索场景：查看人物、物品卡片，点击获取详情。
4. 获得总结：场景完成后，显示学习总结与推荐。

8.2 第一阶段：Streamlit卡片式界面

· 布局：左右两列，左侧场景信息，右侧人物/物品卡片列表。
· 题目区域：使用st.radio或st.text\_input。
· 反馈区域：显示评语和解析。
