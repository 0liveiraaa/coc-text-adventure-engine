# COC项目规范符合性审查与修改建议

## 1. 审查范围
- 规范文档: docs/spec_v2_simplified.md
- 核心实现: src/engine, src/agent, src/rule, src/data
- 配置数据: config/world

## 2. 总体结论
当前实现已具备基础模块骨架（Input/DM/Rule/StateEvolution/IO/CLI），但与V2规范仍存在多处关键偏差。尤其是世界配置机制、结局配置入口、模块接口一致性和流程闭环能力，尚不足以稳定支持“完整可通关流程”。

## 3. 主要不符合项（按优先级）

### P0: 会影响核心流程正确性的缺陷

1) GameEngine 调用 DM Agent 的接口不匹配
- 位置: src/engine/game_engine.py:377
- 现状: 传入 DMAgentInput 对象 `self.dm_agent.parse_intent(dm_input)`
- 对照实现: src/agent/dm_agent.py:184 的 `parse_intent` 签名要求 `player_input: str, game_state: Optional[GameState], dialogue_history`
- 风险: 运行时参数语义错误，导致DM解析上下文缺失或行为异常。
- 建议: 统一为强类型接口。推荐在 `DMAgent` 增加 `parse_intent_structured(input: DMAgentInput, game_state: GameState)`，并在引擎侧仅调用这一入口。

2) GameEngine 调用 IO 的变更应用接口不存在
- 位置: src/engine/game_engine.py:463
- 现状: 调用 `self.io.apply_state_change(change)`
- 对照实现: src/data/io_system.py 仅提供 `apply_changes(changes)` 和 `_apply_single_change(change)`
- 风险: Step 6 无法执行，状态无法落库，流程中断。
- 建议: 二选一并统一：
  - 在 `IOSystem` 新增 `apply_state_change(change)` 单条接口；
  - 或引擎改为批量调用 `apply_changes`。

3) State Evolution 类名与引擎导入不一致
- 位置: src/engine/game_engine.py:25, src/agent/state_evolution.py:92
- 现状: 引擎导入 `StateEvolutionAgent`，实现中类名是 `StateEvolution`
- 风险: 模块初始化失败。
- 建议: 统一命名（建议保留 `StateEvolutionAgent`，与系统角色语义一致），并同步更新导入与实例化处。

4) COC 大成功阈值实现错误
- 位置: src/rule/rule_system.py:67
- 现状: `critical_threshold = min(1, target_value // 5)`，几乎恒为1
- 规范: “骰子 == 1 或 <= 目标值/5（取较小）”。
- 风险: 大成功概率严重失真，规则系统不符合规范。
- 建议: 改为显式规则函数，例如 `critical_threshold = max(1, target_value // 5)` 并结合是否采用“取较小”版本做参数化；同时补单元测试覆盖边界值。

### P1: 与规范流程存在明显偏差

5) 回合开始未通过 IO 统一清空 current_event
- 位置: src/engine/game_engine.py:345
- 现状: 仅清空内存对象 `char.memory.clear_current()`
- 规范: Step 1 应由 IO 系统清空并入 log。
- 风险: 内存与存储状态不一致，存档/读档后行为漂移。
- 建议: Step 1 统一调用 `IOSystem.clear_current_events()`，再回读或同步内存。

6) GameContext 过度简化，不满足规范输入需求
- 位置: src/engine/game_engine.py:578
- 现状: 仅计数类字段（characters_count/items_count/maps_count）
- 规范: DM 与状态推演需要 location、nearby_characters、nearby_items、player_status 等实体级信息。
- 风险: LLM决策质量低，难以实现稳定叙事与变更。
- 建议: 复用/抽象 `dm_agent.py` 与 `state_evolution.py` 内的上下文构建逻辑，统一一份 `ContextBuilder`。

7) 基础指令集未覆盖规范清单，且存档读档是占位实现
- 位置: src/agent/input_system.py:52, src/agent/input_system.py:482, src/agent/input_system.py:530
- 现状: 缺少 `\use`, `\give`, `\reset`, `\debug`；`\save`/`\load`无真实闭环。
- 风险: CLI 功能与规范不一致，调试与流程验证困难。
- 建议: 将命令执行分层：Input 仅解析，实际副作用由 GameEngine 统一调度并落库。

8) WorldLoader 对 player_id/current_scene/turn_order 使用硬编码
- 位置: src/data/init/world_loader.py:69, src/data/init/world_loader.py:70, src/data/init/world_loader.py:71
- 现状: 固定 `char-player-01`、`map-room-library-01`，turn_order=所有角色键顺序。
- 风险: 多剧本不可复用，NPC回合顺序不可配置。
- 建议: 改为读取世界元配置（world manifest）中的 `player_id`, `start_map_id`, `turn_order_policy`。

### P2: 架构一致性与可维护性问题

9) 世界配置仍是“大表JSON”，不利于模块化维护
- 位置: src/data/init/world_loader.py:83（读取 characters.json）及对应 items/maps
- 现状: 角色/物品/地图均为单文件聚合。
- 风险: 协作冲突高、局部复用差、难支持多世界并行开发。
- 建议: 采用目录化拆分（见第4节方案）。

10) 结局条件配置入口缺失，仍在代码硬编码
- 位置: src/engine/game_engine.py:82
- 现状: `self.end_condition` 写死在代码。
- 风险: 不同世界无法声明不同胜负条件和结局文本策略。
- 建议: 在世界配置增加 `endings/` 与 `rules.json`，引擎启动时注入。

11) LLM 配置方式未产品化，且包含“注释式待办”
- 位置: src/agent/llm_service.py:88, src/agent/llm_service.py:90
- 现状: 仅环境变量；代码行内出现 `###改为...`。
- 风险: 部署与调试成本高，注释污染可读性。
- 建议: 支持优先级配置链：`config/llm.json` > 环境变量 > 默认值，并去除行内待办注释。

## 4. 针对你提出的“世界配置机制”改造方案（建议作为第一优先级）

建议采用“多世界目录 + 清单驱动”结构：

```text
config/
  world/
    mysterious_library/
      world.json                # 世界清单（入口）
      rules.json                # 回合规则、难度映射、全局参数
      endings/
        ending_main_clear.json
        ending_bad_san_loss.json
      characters/
        char-player-01.json
        char-guard-01.json
      items/
        item-key-01.json
        item-book-01.json
      maps/
        map-room-library-01.json
        map-room-corridor-01.json
      prompts/
        dm_system_prompt.md
        state_evolution_prompt.md
```

`world.json` 建议字段：
- `world_id`, `world_name`
- `player_id`
- `start_map_id`
- `turn_order`（显式列表）或 `turn_order_policy`
- `entry_scene_narrative`
- `end_condition_summary`
- `enabled_modules`（可选）
- `data_index`（角色/物品/地图文件索引，可选）

`endings/*.json` 建议字段：
- `id`, `priority`, `condition_expr`, `end_narrative`, `is_bad_ending`

说明：
- `condition_expr` 可先支持简单DSL（例如 `player.status.hp<=0`）或规则函数名映射。
- 状态推演系统只负责“建议触发的ending_id”，最终由引擎统一判定，避免LLM直接控制结局真值。

## 5. 代码层面的落地步骤（建议迭代）

1) 新增世界清单与目录化加载器
- 新建 `WorldManifest` 模型和 `WorldRepository`。
- `WorldLoader` 改为扫描目录并逐文件装载实体。

2) 统一接口与上下文构建
- 修正 DM/State/IO 接口不一致问题（P0项）。
- 新增 `ContextBuilder`，由引擎统一生成模块输入。

3) 结局系统产品化
- 新增 `EndingEvaluator`：接收 `game_state + endings`，输出 `is_end/end_narrative`。
- 引擎每回合 Step 6 后调用 evaluator，覆盖LLM输出中的结局布尔值。

4) 指令系统补齐规范
- 补 `\use`, `\give`, `\reset`, `\debug`。
- `\save/\load` 接入 `engine.save_game/load_game` 真实逻辑。

5) 质量保障
- 为规则系统（阈值边界）和IO变更应用增加单元测试。
- 增加“完整通关最小剧本”回归测试（从新档到结局）。

## 6. 额外建议（除你提出外）

- 建议把“LLM产生的变更”与“规则引擎允许的变更白名单”结合，避免越权修改关键字段（如直接改 player_id）。
- 建议在 `StateChange.field` 增加校验器，限制非法路径与危险路径。
- 建议在CLI增加 `\debug trace on/off`，打印每步输入输出（可脱敏）。
- 建议把 `description.hint` 在日志和UI输出层做统一脱敏，防止误泄露给玩家。

## 7. 与规范一致性简评

已基本对齐：
- 模块分层方向正确。
- 数据模型包含 public/hint、memory/current_event。

未对齐重点：
- 世界配置与结局配置机制。
- 引擎7步流程中的IO一致性。
- 模块接口统一性与可运行闭环。

---

若以“最快达到可通关”作为目标，建议先完成：
- P0全部修复
- 世界目录化 + 结局配置入口
- 一个最小可通关剧本的自动回归测试
