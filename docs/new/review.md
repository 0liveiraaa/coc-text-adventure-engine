- 规则1: 鉴定系统输出为事实锚点，不可被导演/推演/合并链路改写
- 规则2: 同回合对外展示以合并叙事为唯一广播文本

## 1. 已确认的问题

### 1.1 回合数不增加 (P0)

**问题描述**: 玩家输入行动后，回合数没有正常增加，始终停留在第1回合。

**根本原因**: `StateEvolution` 返回的 `resolved` 字段默认为 `false`，导致 `GameEngine._turn_end()` 方法不增加回合数。

**代码位置**:
- `src/engine/game_engine.py` 第376-379行
- `src/agent/state_evolution.py` 第687-733行

**当前逻辑**:
```python
def _turn_end(self, resolved: bool = True):
    if resolved:
        self.game_state.turn_count += 1  # 只有resolved=True时才增加
    else:
        logger.debug("回合未解决，保持当前行动者")  # resolved=False时不增加
```

**影响**: 影响游戏进程跟踪、回合制机制、叙事上下文的时间线。

**修复建议**: 
1. 修改提示词，明确大多数行动应返回 `resolved=true`
2. 或修改 `_turn_end` 方法，强制每轮都增加回合数

### 1.2 纯对话无法触发NPC响应 (P0)

**问题描述**: 玩家输入"和守卫说话"等对话类指令时，NPC不会做出回应。

**根本原因**: 当 `is_dialogue=true` 时，`GameEngine` 直接返回 DM Agent 的回复，不执行后续的 NPC 响应流程。

**代码位置**:
- `src/engine/game_engine.py` 第376-379行

**当前逻辑**:
```python
if dm_output.is_dialogue:
    result["response"] = dm_output.response_to_player
    self._record_dm_dialogue(natural_input, dm_output.response_to_player)
    return result  # <-- 直接返回，不执行NPC响应
```

**影响**: 玩家无法与NPC进行基础对话，影响游戏交互体验。

**修复建议**:
1. 修改 `GameEngine` 逻辑：即使 `is_dialogue=true`，如果有 `npc_response_needed=true`，仍需执行NPC响应流程
2. 修改 DM Agent 提示词：明确纯对话时仍需设置 `npc_response_needed=true`

**修复后的逻辑**:
```python
if dm_output.is_dialogue:
    if dm_output.npc_response_needed and dm_output.npc_actor_id:
        # 继续执行NPC响应流程，不返回
        pass
    else:
        # 没有NPC响应，直接返回
        result["response"] = dm_output.response_to_player
        return result
```

### 1.6 读档未恢复世界级配置，导致规则漂移 (P1)

### 1.7 变更失败仅记录日志，未阻断流程导致叙事与状态分叉 (P1)

**问题描述**: 当 `io.apply_state_change()` 返回非0错误码时，系统仅打印 warning 并继续流程，不会反馈给上游或中止当前回合。

**根本原因**:
- `_apply_changes()` 只做逐条日志记录，不聚合失败、不回滚、不改写 `result["success"]`
- `StateEvolution.validate_changes()` 对字段路径仅做“非空检查”，无法提前发现大量无效字段

**代码位置**:
- `src/engine/game_engine.py` 第773-790行 (`_apply_changes`)
- `src/agent/state_evolution.py` 第849-853行 (`validate_changes` 字段路径检查)

**影响**:
- 玩家看到叙事“已发生”，但状态未落地，形成隐式数据分叉
- 多条变更部分成功时，可能引入难复现的中间态

**修复建议**:
1. `_apply_changes()` 返回失败列表并由 `process_input()` 统一处理（降级/重试/中止）
2. 对同一轮变更采用“全成功才提交”的事务式策略（至少在内存态实现原子性）
3. 在 `validate_changes()` 增加字段路径可达性检查（含嵌套字段）

## 2. 设计决策澄清

### 2.1 叙事系统架构

**澄清**: 叙事系统分为三个层次，各自有不同的处理方式：

| 层次 | 功能 | 是否使用LLM | 说明 |
|------|------|-------------|------|
| **叙事生成** | StateEvolution生成玩家/NPC行动结果叙事 | ✅ 是 | 每个行动者独立生成叙事片段 |
| **叙事合并** | 将多个独立叙事合并为完整连贯的全局叙事 | ✅ **是** | 解决冲突、统一视角 |
| **上下文压缩** | 历史事件超出窗口时的压缩处理 | ❌ 否 | 使用滚动窗口机制，简单截断 |

**关键原则**: 合并后的全局叙事必须通过 `NarrativeContext.get_context_for_llm()` 提供给所有LLM模块作为上下文提示词。

---

### 2.2 NPC决策分工

**澄清**: NPCDirector 和 StateEvolution 的分工：

| 模块 | 职责 | 输出 |
|------|------|------|
| **NPCDirector** | 决定NPC行动意图 | `NPCActionForm`（行动类型、目标、意图描述） |
| **StateEvolution** | 执行NPC行动，生成具体叙事 | `StateEvolutionOutput`（叙事、状态变更） |

**StateEvolution中的NPC推演指南需要保留**，但角色从"决定NPC做什么"变为"根据NPCDirector的决定，生成NPC怎么做"。

### 2.3 玩家-NPC对话支持

**澄清**: 系统已支持玩家与NPC对话，但存在实现缺陷：

- ✅ `is_dialogue` 字段可识别纯对话

- ❌ **缺陷**: `is_dialogue=true` 时直接返回，不触发NPC响应

## 5. 根本问题分析（跨问题）

> 本节从所有已确认问题中提炼系统性根因，而非针对单点bug的归因。
> 核心结论：系统模块分层已较完整，但**跨模块契约、领域不变量、失败语义**三类机制未工程化为硬约束，导致行为在边界处漂移。

### 5.1 LLM输出与代码执行之间的契约边界模糊

**涉及问题**: 0.3/0.4/0.6/1.1/1.2

**问题本质**:
系统采用"LLM决策 + 代码执行"的混合架构，但两者之间的**语义契约**未明确定义和强制校验：

| 契约缺陷 | 具体表现 | 后果 |
|---------|---------|------|
| **字段语义模糊** | `resolved`字段由LLM输出，但代码依赖其决定回合推进；`is_dialogue`与`npc_response_needed`组合控制NPC响应流程 | 流程控制受LLM输出波动影响 |
| **类型契约缺失** | LLM可能返回list而非string、返回嵌套列表污染实体集合 | 运行时类型错误、数据污染 |
| **状态机未形式化** | 回合推进、NPC响应触发等关键流程依赖多字段组合判断，无统一状态机定义 | 控制流存在早退分叉，难以测试 |

**根因诊断**:
- 关键流程控制字段（`resolved`/`is_dialogue`/`npc_response_needed`）的语义分散在Prompt约定与代码分支中
- 缺少"契约层"统一声明：字段取值范围、合法组合、驱动规则
- LLM输出校验停留在"字段存在性"，未覆盖"字段语义正确性"

### 5.2 领域不变量未下沉为数据层强约束

**涉及问题**: 0.4/1.4/1.5/1.7

**问题本质**:
系统设计为"字段写入引擎"而非"领域操作引擎"，核心业务不变量依赖调用方自觉维护：

| 不变量类型 | 当前状态 | 期望状态 |
|-----------|---------|---------|
| **位置唯一性** | 物品可同时出现在多个位置，无校验 | 物品只能处于唯一位置 |
| **持有关系双向一致** | `item.location`与`character.inventory`独立维护 | 任一变更自动同步另一端 |
| **数值边界** | HP可超过max_hp，SAN可低于0 | 自动clamp到合法范围 |
| **实体集合完整性** | ADD操作可能引入嵌套列表 | 强制扁平化、去重、类型断言 |

**根因诊断**:
- `IOSystem.apply_changes()`只验证单条变更的"语法合法性"（字段路径是否存在）
- 未验证"语义合法性"（变更后整体状态是否满足不变量）
- 领域操作（如物品转移）未封装为原子原语，需调用方手动展开为多条变更

### 5.3 变更执行缺少事务语义，失败路径不可控

**涉及问题**: 0.5/1.7

**问题本质**:
状态变更采用"尽力而为"（best-effort）模式，无事务边界、无失败聚合、无回滚机制：

```
当前行为: [变更1成功] → [变更2失败(仅日志)] → [变更3成功] → 继续流程
期望行为: [变更1成功] → [变更2失败] → 回滚变更1 → 返回失败 → 中止/降级
```

**具体表现**:
- `_apply_changes()`逐条执行，失败仅记录warning，不阻断流程
- 变更结果不聚合返回，主流程无法感知"部分成功"
- 叙事已生成但状态未落地，形成"叙事-状态分叉"

**根因诊断**:
- 变更执行层职责边界不清：只负责"尝试执行"，不负责"结果承诺"
- 主流程缺少失败策略协议：中止？降级？重试？补偿？
- 异常路径可观测性不足：静默继续导致问题难以追踪

### 5.4 运行态配置与持久化快照割裂

**涉及问题**: 1.6

**问题本质**:
引擎行为依赖两类配置：**游戏状态**（GameState）与**世界参数**（world_name/end_condition/npc_mode等），但存档只持久化前者：

| 配置类型 | 存档覆盖 | 读档恢复 | 后果 |
|---------|---------|---------|------|
| GameState | ✅ | ✅ | 正常 |
| world_name | ❌ | ❌ | 读档后可能切换到错误世界 |
| end_condition | ❌ | ❌ | 结局判定规则漂移 |
| npc_response_mode | ❌ | ❌ | NPC行为策略不一致 |
| LLM开关 | ❌ | ❌ | 叙事合并策略突变 |

**根因诊断**:
- 存档模型与运行态依赖不匹配：存的是"数据"，丢的是"行为配置"
- 缺少"最小可复现运行快照"定义：哪些字段决定行为、持久化顺序、重建步骤
- `load_game()`后未调用`apply_world_settings()`对齐运行态

### 5.5 异常处理机制缺失，错误传播不可控

**涉及问题**: 0.1/0.2/0.3/0.5

**问题本质**:
系统在多处采用"打印错误+继续执行"模式，缺少熔断、降级、恢复机制：

| 异常场景 | 当前行为 | 风险 |
|---------|---------|------|
| CLI渲染异常 | 打印错误，进入下一轮循环 | 异常刷屏 |
| 编码异常 | 崩溃 | 进程终止 |
| 类型异常 | 打印错误，继续 | 数据污染累积 |
| 变更执行异常 | 记录日志，继续 | 叙事-状态分叉 |

**根因诊断**:
- 异常处理策略不统一：有的崩溃、有的静默、有的刷屏
- 缺少"错误边界"定义：哪些错误可恢复、哪些必须熔断
- 无降级策略：异常后如何优雅降级而非硬崩溃

### 5.6 测试覆盖偏主路径，边界条件验证不足

**涉及问题**: 全部问题

**问题本质**:
回归测试主要覆盖"功能点"，未系统化覆盖"契约边界"和"异常路径"：

| 测试类型 | 当前覆盖 | 本次暴露的缺口 |
|---------|---------|---------------|
| 功能测试 | ✅ 较完整 | - |
| 契约测试 | ❌ 缺失 | 字段语义、字段组合规则 |
| 不变量测试 | ❌ 缺失 | 多变更后全局一致性 |
| 异常路径测试 | ❌ 缺失 | 部分失败、配置丢失、类型漂移 |
| 跨模块集成测试 | ⚠️ 部分 | 模块交界处的状态传递 |

**根因诊断**:
- 测试策略以"功能是否实现"为导向，未以"契约是否遵守"为导向
- 缺少不变量断言：每次变更后校验全局一致性
- LLM输出模拟不足：未覆盖异常形态（类型错误、字段缺失、语义冲突）

---

### 5.7 根因归纳与治理优先级

| 根因类别 | 核心问题 | 治理优先级 | 治理方向 |
|---------|---------|-----------|---------|
| **契约模糊** | LLM输出与代码执行语义边界不清 | P0 | 建立契约层：字段语义声明 + 状态机定义 + 输出校验 |
| **不变量缺失** | 领域约束依赖调用方自觉 | P0 | 下沉不变量：数据层强校验 + 领域操作原语 |
| **事务缺失** | 变更失败静默继续 | P1 | 引入事务语义：失败聚合 + 回滚/降级策略 |
| **配置割裂** | 存档不完整导致行为漂移 | P1 | 统一快照契约：持久化范围 + 重建步骤 |
| **异常处理** | 错误传播不可控 | P2 | 定义错误边界：熔断策略 + 降级方案 |
| **测试缺口** | 边界条件验证不足 | P2 | 补充三类测试：契约/不变量/异常路径 |

> **核心结论**: 当前问题并非单点实现失误，而是**契约未硬化 + 不变量未中心化 + 失败未事务化**的组合结果。修复时应优先做机制性改造，避免在每个症状点继续打补丁。

## 6. 改进计划

> 目标不是逐个修补表象，而是把“契约、状态、不变量、失败语义”四个基础层先立住，再向上恢复功能与体验。

### 6.1 阶段一：先收紧契约，冻结高风险流程

**目标**:
- 消除回合推进、NPC响应、鉴定结果显示等关键流程的语义歧义
- 让 `resolved`、`is_dialogue`、`npc_response_needed` 这类字段只承担一种明确职责

**执行项**:
1. 明确 `state_evolution_prompt.md` 中 `resolved` 的取值规则
2. 明确 `system_prompt.md` 中 `is_dialogue` 与 `npc_response_needed` 的组合规则
3. 在 `GameEngine` 内把关键流程判断收拢到统一分支，避免多处早退
4. 为关键字段增加单元测试和契约测试

**验收标准**:
- 纯对话场景可稳定触发或不触发 NPC 响应，行为由字段组合决定且可测试
- 每轮行动后的回合推进行为稳定，不再依赖隐式默认值
- 关键提示词与代码逻辑一致，不再出现“提示词说一种、代码做另一种”

### 6.2 阶段二：下沉领域不变量，封装核心领域操作

**目标**:
- 把物品、角色、地图之间的核心一致性从“调用方约定”变成“系统强约束”
- 让物品转移、持有关系同步、数值边界收敛成为统一原语

**执行项**:
1. 实现 `ConsistencyChecker`，集中检查 `location` / `inventory` / 唯一性
2. 为物品移动增加自动同步逻辑，禁止只改单边字段
3. 为 HP、SAN、实体列表等补充范围与结构化约束
4. 将“物品转移”收敛为领域原语，避免上层手写多条变更

**验收标准**:
- 任意一组变更提交后，系统能够发现并拒绝不一致状态
- 只改 `location` 时，持有关系可以自动同步或被显式拒绝
- 嵌套列表、重复实体、非法数值能在进入持久层前被拦截

### 6.3 阶段三：建立失败语义，补齐事务边界

**目标**:
- 让失败变成主流程可感知的结果，而不是静默日志
- 避免“叙事已发生、状态没落地”的分叉

**执行项**:
1. 改造 `_apply_changes()`，返回失败集合而不是只写日志
2. 在 `process_input()` 中统一决定中止、降级或重试
3. 为同一回合变更引入全成功提交策略，至少保证内存态原子性
4. 将字段可达性校验前移到 `validate_changes()`

**验收标准**:
- 单条变更失败时，主流程能明确感知并执行约定策略
- 回合叙事与状态更新不会长期分叉
- 测试中可稳定复现“失败后中止/降级”行为

### 6.4 阶段四：统一存档快照，修复运行态漂移

**目标**:
- 让存档成为可复现运行态，而不是只保存局部状态
- 保证读档后世界配置、规则、提示词策略与存档时一致

**执行项**:
1. 存档增加 world-level metadata
2. 读档时先恢复 metadata，再调用 `apply_world_settings()`
3. 增加跨世界、跨配置的读档回归测试
4. 记录存档 schema version，预留向后兼容空间

**验收标准**:
- 读档后 `world_name`、结局规则、NPC响应模式与存档时一致
- 配置切换后不会回退到默认值
- 老存档能被识别并安全迁移或降级处理

### 6.5 阶段五：补齐异常处理与测试护栏

**目标**:
- 把异常控制从“临时容错”提升为“稳定护栏”
- 让边界条件、异常路径和集成契约有持续回归能力

**执行项**:
1. 定义异常边界：哪些错误要熔断，哪些可以降级
2. CLI 保持防御性输出，避免刷屏和编码崩溃反复出现
3. 补充契约测试、不变量测试、异常路径测试
4. 对 LLM 输出增加异常形态样本测试，包括类型漂移和字段冲突

**验收标准**:
- 异常不会无限刷屏，也不会悄悄污染后续流程
- 测试能覆盖已知问题的主路径和边界路径
- 新增功能必须同时补充对应契约测试

### 6.6 推荐推进顺序

1. 先做阶段一，锁定流程契约，避免继续扩大语义漂移
2. 再做阶段二，保证数据不变量，防止修复后继续产生脏数据
3. 接着做阶段三，确保失败可感知、可中止、可回滚
4. 然后做阶段四，避免读档和跨世界配置把前面修复全部冲掉
5. 最后做阶段五，把这些规则固化成测试与异常护栏

> 执行原则: 先机制，后症状；先约束，后扩展；先回归，后优化。
