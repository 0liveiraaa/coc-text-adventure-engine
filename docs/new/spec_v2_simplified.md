# COC文字冒险游戏框架Spec文档 V2


---

## 1. 概述

### 1.1 项目目标

构建一个以LLM为主要驱动、基于COC规则的简化版文字冒险游戏框架。通过大幅精简原设计的复杂系统，实现低成本、高自由度的游戏体验。

### 1.2 设计

#### 1.2.1 低成本高自由度

框架追求最小化硬编码逻辑，将游戏叙事的自由度最大化地交给AI处理。通过简化系统降低开发和维护成本，同时保持足够的鲁棒性支持涌现式叙事。

#### 1.2.2 数据流双轨制

```
文字数据 <----LLM----> 数值数据
```

- **文字数据**: 玩家的自然语言输入、AI生成的叙事描述
- **数值数据**: 角色属性、物品状态、位置信息等结构化数据
- **LLM职责**: 负责两种数据之间的转换——理解玩家意图并转化为数值操作，将数值变化解释为生动的文字描述

## 2. 核心架构

### 2.1 简化后的游戏流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           一轮游戏完整流程                                   │
└─────────────────────────────────────────────────────────────────────────────┘

    ┌──────────┐      ┌──────────┐      ┌──────────┐      ┌──────────┐
    │ 玩家输入 │      │  数据库  │      │ IO系统   │      │ Input系统│
    └────┬─────┘      └────┬─────┘      └────┬─────┘      └────┬─────┘
         │                 │                 │                 │
         │ 自然语言指令    │                 │                 │
         └────────────────>│                 │                 │
                           │ 读取游戏状态    │                 │
                           └────────────────>│                 │
                                             │ 游戏数据        │
                                             └────────────────>│
                                                               │
                          ┌────────────────────────────────────┘
                          ▼
                   ┌────────────┐
                   │  DM Agent  │
                   │ - 输入解析 │
                   │ - 意图识别 │
                   │ - 鉴定判断 │
                   └─────┬──────┘
                         │ 结构化意图
                         │ (是否需要鉴定?
                         │  鉴定类型/属性/目标)
                         ▼
                   ┌────────────┐
                   │  规则系统  │
                   │ COC鉴定计算 │
                   └─────┬──────┘
                         │ 鉴定结果
                         │ (大成功/成功/失败/大失败)
                         ▼
                   ┌────────────┐
                   │ 状态推演系统│
                   │ - 生成叙事 │
                   │ - 状态变更 │
                   │ - NPC扮演  │
                   └─────┬──────┘
                         │ 变更表单
                         │ (update/add/del列表)
                         │ + 是否解决标识
                         ▼
                   ┌────────────┐
                   │   IO系统   │
                   └─────┬──────┘
                         │ 写入变更
                         ▼
                   ┌────────────┐
                   │   数据库   │
                   └────────────┘
```

### 2.2 各模块职责说明

| 模块 | 核心职责 | AI决策权 |
|------|----------|----------|
| **IO系统** | 统一的数据读写接口，隔离数据库操作细节 | 低 |
| **Input系统** | 接收并初步处理玩家输入，区分指令和自然语言 | 低 |
| **DM Agent** | 解析玩家意图，判断是否需要鉴定，确定鉴定参数 | **高** |
| **规则系统** | 执行COC规则的数值计算，输出鉴定结果 | 低（纯计算） |
| **状态推演系统** | 根据鉴定结果推演世界变化，生成叙事，更新状态，兼容NPC扮演 | **高** |

## 3. 数据层设计

### 3.1 二级描述系统

简化后的描述系统仅保留两个层级：

#### 3.1.1 public（公开描述）

可见的文字描述，可被AI更改，实现AI输出的持久化。

```json
{
  "public": [
    {
      "description": "一段可见的描述文本"
    },
    {
      "description": "AI生成的新增描述"
    }
  ]
}
```

**用途**:
- 玩家可见的场景描述
- 角色外貌特征
- 物品外观信息
- AI可以随时追加、修改

#### 3.1.2 hint（开发者提示）

仅LLM知道的开发者提示，用于提升扮演效果，任何时候不得外泄给玩家。

```json
{
  "hint": "这是一个隐藏的线索，只有DM知道。当玩家调查书架时，可以暗示这里有暗门。"
}
```

**用途**:
- 给AI的扮演指导
- 隐藏的背景信息
- 剧情伏笔提示
- NPC的性格暗示

### 3.3 数据存储方案

- **格式**: JSON静态文件
- **结构**: 每个实体类型独立文件，按ID索引
- **初始数据**: `config/world/` 目录存放初始配置
- **运行时数据**: 运行时加载到数据库，定期保存到`data/saves/`下的存档文件夹里

### 4.2 Input系统

**职责**: 接收玩家输入，区分基础指令和自然语言交互。

#### 4.2.1 输入处理流程

```
玩家输入
    │
    ├─→ 基础指令（以\开头）→ 直接处理 → 输出结果
    │
    └─→ 自然语言 → 传递给DM Agent
```

#### 4.2.2 基础指令清单

| 指令 | 功能 | 处理方式 |
|------|------|----------|
| `\look` | 查看当前场景 | 直接查询GameState |
| `\inventory` | 查看背包 | 直接查询GameState |
| `\pickup <物品名>` | 捡起物品 | 直接更新状态 |
| `\drop <物品名>` | 放下物品 | 直接更新状态 |
| `\help` | 显示帮助 | 静态文本 |
| `\save` | 保存进度 | IO系统 |
| `\load` | 加载进度 | IO系统 |
| `\exit` | 退出游戏 | 系统控制 |

### 4.3 DM Agent

**职责**: 解析玩家输入，理解意图，判断鉴定需求。

**设计原则**: 将判断权力交给AI，不预设复杂规则。

#### 4.3.1 输入格式

```json
{
  "system_prompt": "初始提示词，定义角色和任务",
  "player_input": "玩家输入的自然语言",
  "dialogue_history": ["之前的对话记录"],
  "game_context": {
    "current_location": "当前地图数据",
    "nearby_characters": "附近角色列表",
    "nearby_items": "附近物品列表",
    "player_status": "玩家状态"
  }
}
```

#### 4.3.2 输出格式

```json
{
  "is_dialogue": false,
  "response_to_player": "给玩家的回复文本（当is_dialogue为true时）",
  "needs_check": true,
  "check_type": "对抗鉴定",
  "check_attributes": ["侦查"],
  "check_target": "char-guard-01",
  "action_description": "玩家试图悄悄绕过守卫"
}
```

#### 4.3.3 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_dialogue` | bool | 是否为纯对话，true时不进入规则系统 |
| `response_to_player` | string | 给玩家的回复，仅在is_dialogue=true时使用 |
| `needs_check` | bool | 是否需要COC鉴定 |
| `check_type` | string | "对抗鉴定" 或 "非对抗鉴定" |
| `check_attributes` | list | 鉴定使用的属性列表，如["侦查"], ["力量", "体格"] |
| `check_target` | string | 对抗鉴定的目标ID，非对抗时为null |
| ` dificulty ` | string | 非对抗鉴定时的难度 |
| `action_description` | string | 行动的自然语言描述，传递给状态推演系统 |

### 4.4 规则系统

**职责**: 执行COC规则的数值计算。

**特点**: 纯计算逻辑，无AI参与。

#### 4.4.1 鉴定类型

1. **非对抗鉴定**: 角色属性 vs 难度值
2. **对抗鉴定**: 角色A属性 vs 角色B属性

#### 4.4.2 鉴定结果判定

COC标准判定规则:
难度等级	             判定阈值	           适用场景	      示例（技能值 60）
常规（Regular）	技能 / 属性原值	普通难度，正常可完成的动作	≤60 成功
困难（Hard）	技能 / 属性值 ×1/2（向下取整）	有挑战，需额外努力的动作	≤30 成功
极难（Extreme）	技能 / 属性值 ×1/5（向下取整）	接近人类极限，成功率低	≤12 成功
对抗检定	双方同技能 / 属性对比成功等级	玩家与 NPC / 其他玩家对抗	困难成功 > 常规成功，极难 > 困难

```
判定结果 = d100骰子结果 vs 目标值

- 大成功: 骰子结果 == 1 或 骰子结果 <= 目标值/5 (取较小)
- 成功: 骰子结果 <= 目标值
- 失败: 骰子结果 > 目标值
- 大失败: 骰子结果 == 100 或 骰子结果 >= 96+ (当目标值<50时)
```

#### 4.4.3 输入输出

**输入**:
```json
{
  "check_type": "非对抗鉴定",
  "attributes": ["侦查"],
  "actor_id": "char-player-01",
  "target_id": null,
  "difficulty": "Regular"
}
```

**输出**:
```json
{
  "result": "成功",
  "dice_roll": 45,
  "target_value": 60,
  "actor_value": 60
}
```

### 4.5 状态推演系统

**职责**: 
- 根据鉴定结果推演世界变化
- 生成叙事描述
- 生成状态变更列表
- **兼容NPC扮演**（替代原独立NPC Agent）

**设计原则**: 这是AI权力最大的模块，负责将数值结果转化为世界变化。

#### 4.5.1 输入格式

```json
{
  "system_prompt": "初始提示词",
  "end_condition": "结局条件",
  "check_result": {
    "check_type": "非对抗鉴定",
    "result": "成功",
    "dice_roll": 45,
    "actor_id": "char-player-01",
    "target_id": null,
    "attributes": ["侦查"]
  },                                                           
  "action_description": "玩家试图悄悄绕过守卫",
  "game_context": {
    "actor": "执行者数据",
    "target": "目标数据",
    "location": "当前场景数据",
    "nearby_entities": "附近实体"
  },
  "is_npc_action": false,
  "npc_intent": null,
  "npc_infor" : "要扮演的npc的信息"
}
```

**NPC扮演模式**: 当`is_npc_action=true`时，check_result等应为空,系统根据NPC的current_event和记忆生成行动意图和变更.

#### 4.5.2 输出格式

```json
{
  "narrative": "你成功地避开了守卫的视线，悄悄溜进了侧门。",
  "changes": [
    {
      "id": "char-player-01",
      "field": "location",
      "operation": "update",
      "value": "map-room-library-02"
    },
    {
      "id": "char-guard-01",
      "field": "memory.current_event",
      "operation": "update",
      "value": "你感觉刚才好像有什么动静..."
    }
  ],
  "resolved": true,
  "next_action_hint": null,
  "is_end":" 一个bool值方便游戏确定是否结束",
  "end_narrative":"结局描述字段"
}
```

#### 4.5.3 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `narrative` | string | 生成的叙事文本，将写入相关角色的current_event |
| `changes` | list | 状态变更列表，格式见第5章 |
| `resolved` | bool | 此回合是否已解决，true则进入下一轮 |
| `next_action_hint` | string/null | 如需连续行动，提供下轮行动的提示 |

#### 4.5.4 NPC扮演集成

状态推演系统同时承担原NPC Agent的职责：

- **输入**: NPC的current_event、记忆、状态
- **处理**: LLM根据角色设定生成行动
- **输出**: 与玩家行动相同的结构化格式

这意味着NPC和玩家使用同一套处理流程，真正实现"主体客体平等"。

## 5. 状态变更系统（简化版）

### 5.1 输入格式

变更列表，每个元素为变更对象：

```json
[
  {
    "id": "实体ID",
    "field": "字段路径，支持点分，如attributes.hp",
    "operation": "update/add/del",
    "value": "新值"
  }
]
```

### 5.2 变更类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `update` | 更新字段值 | 修改HP、改变位置 |
| `add` | 向数组添加元素 | 添加物品到背包 |
| `del` | 删除字段或数组元素 | 移除物品、删除状态 |

### 5.3 错误码定义

| 错误码 | 含义 | 处理建议 |
|--------|------|----------|
| `0` | 成功 | 无需处理 |
| `1` | ID不存在 | 检查ID拼写，或先创建实体 |
| `2` | 字段不存在 | 检查字段路径，或使用add创建 |
| `3` | 变更类型错误 | 检查operation字段 |
| `4` | 其他错误 | 记录日志，跳过此变更 |

### 5.4 特殊变更约定

#### 5.4.1 current_event更新

广播系统简化为直接向角色的`memory.current_event`字段写入：

```json
{
  "id": "char-player-01",
  "field": "memory.current_event",
  "operation": "update",
  "value": "你听到身后传来脚步声..."
}
```

每轮开始时，由IO系统自动清空所有角色的current_event,并将其压入log。

## 6. 数据流说明

### 6.1 完整的一轮游戏流程

```
┌─────────────────────────────────────────────────────────────────┐
│ Step 1: 回合开始                                                 │
├─────────────────────────────────────────────────────────────────┤
│ • IO系统清空所有角色的current_event                              │
│ • 确定当前行动者（从行动顺序列表中取出）                          │
│ • 判断是玩家回合还是NPC回合                                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 2: 获取行动意图                                             │
├─────────────────────────────────────────────────────────────────┤
│ 玩家回合:                                                        │
│   • Input系统接收输入                                            │
│   • 若是基础指令，直接处理并跳转Step 6                            │
│   • 若是自然语言，传递给DM Agent                                  │
│                                                                  │
│ NPC回合:                                                         │
│   • 直接调用状态推演系统（is_npc_action=true）                    │
│   • 由AI根据NPC记忆和状态生成行动                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 3: DM Agent解析                                             │
├─────────────────────────────────────────────────────────────────┤
│ 输出结构化意图:                                                  │
│   • is_dialogue: true → 直接回复玩家，回合结束                    │
│   • needs_check: false → 跳转Step 5                              │
│   • needs_check: true → 继续Step 4                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 4: 规则系统鉴定                                             │
├─────────────────────────────────────────────────────────────────┤
│ • 根据DM Agent提供的参数执行COC鉴定                               │
│ • 计算成功等级（大成功/成功/失败/大失败）                         │
│ • 返回鉴定结果对象                                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 5: 状态推演系统                                             │
├─────────────────────────────────────────────────────────────────┤
│ • 输入：鉴定结果 + 行动描述 + 游戏上下文                          │
│ • AI推演：发生了什么？状态如何变化？                              │
│ • 输出：叙事文本 + 变更列表 + 是否通关                            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 6: 应用状态变更                                             │
├─────────────────────────────────────────────────────────────────┤
│ • IO系统执行变更列表中的每个变更                                  │
│ • 记录错误码，失败变更记录到日志                                  │
│ • 成功的变更立即生效                                             |
| • 系统检查输出是否包含结局如果是就输出结局并给出结局文本           │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ Step 7: 回合结束                                                 │
├─────────────────────────────────────────────────────────────────┤
│ • 将当前行动者移至列表末尾                                        │
│ • 若resolved=false，保持同一行动者（连动机制）                    │
│ • 返回Step 1开始下一轮                                           │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 数据在各模块间的流转

| 阶段 | 数据形式 | 流转路径 |
|------|----------|----------|
| 原始输入 | 自然语言字符串 | Player → Input系统 |
| 结构化意图 | JSON对象 | Input系统 → DM Agent → 规则系统 |
| 鉴定结果 | JSON对象 | 规则系统 → 状态推演系统 |
| 状态变更 | JSON数组 | 状态推演系统 → IO系统 → 数据库 |
| 叙事输出 | 自然语言字符串 | 状态推演系统 → Player（通过current_event） |

## 10. 实现建议

### 10.1 开发优先级

1. **P0（核心）**: IO系统、DM Agent、规则系统、状态推演系统、数据格式

- `\debug` 模式：显示每个模块的输入输出

- 日志导出：导出完整游戏流程用于分析

## 附录A: 错误处理策略

### A.1 LLM输出格式错误

- **场景**: AI返回的JSON格式不正确
- **处理**: 使用retry机制（最多3次）并在请求字段中增加erro提示帮助其进行更改，仍失败则返回友好提示或采用默认回复作为降级策略

### A.2 变更应用失败

- **场景**: 部分变更因ID/字段不存在而失败
- **处理**: 记录错误日志，继续应用其他变更，不中断流程

### A.3 循环检测

- **场景**: AI生成导致无限循环的变更
- **处理**: 设置单轮最大变更数限制（如10个），超限则截断并警告
